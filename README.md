## 展示策略

> 2026-07-16。本目录是《EVSE Pro 使用功能说明书》的权威源；本文记录「如何展示、展示什么」的分析结论，供后续多语言化与发布通道落地时参照。

| 文档 | 导向 | 回答 | 状态 |

## 二、如何展示：一源多出口

**权威源 = 本目录的结构化 Markdown**（章节 + 锚点 + 表格），派生三个出口：

1. **App 内嵌（推荐终态）**——走帮助中心同一通道：server 出 HTML、App WebView 打开、`?lang` 随 App 语言。两种落法二选一：
   - 帮助中心内新增「功能指南」大章（改动最小，目录变长）；
   - 独立 `/user-guide` 页面 + 「我的」页新增「使用指南」行（入口更清晰，需 App 侧加一行 + server 路由）。
   - 增值项：**上下文帮助深链**——App 各页面可带锚点直达对应章节（如设置页 →`/user-guide#settings`），复用现有稳定深链机制。
2. **PDF / 印刷**——从 Markdown 派生 A5 说明书排版，给包装盒、渠道、客服培训（后续需要时再做排版）。
3. **HTML 单页（Artifact）**——内部评审 / 运营 / 客服培训用的可视化版本，已随本次交付。

多语言：权威源先中文定稿 → 英文翻译 → 其余 6 语跟随（与 App 8 语言、帮助中心双语的既有路径一致）。

## 三、展示什么：内容设计的五条原则

1. **按用户旅程组织，不按代码模块**——快速上手 → 日常充电 → 自动化省钱 → 记录 → 分享 → 维护 → 账号；共 13 章 + 2 附录。
2. **四维前提徽标是核心增值**——本 App 每个操作的可用性由四个独立维度决定：登录 🔑 / 近场 🔵 vs 远程 ☁️ / 物主 👤 vs 共享 / 型号能力 🔧。说明书把这四个门禁做成统一图例贯穿全篇，并汇总为附录 A 速查矩阵。这是用户困惑的最大来源（「为什么我这里没有这个按钮」），也是 FAQ 覆盖不了的系统性信息。
3. **文案与 App 逐字一致**——所有「」内文字均来自 `zh-Hans.lproj/Localizable.strings` 真实文案，用户可直接对照界面。状态词以 App 实际显示为准（如「车辆已暂停」而非帮助中心的「车辆已停止」；预约模式用 App 分段名「一次性/循环」）。
4. **诚实红线**（对齐销售手册 §8 对外口径）——未出货不写（RFID/指纹/场景创建=「即将上线」）；估算值标注估算（续航/减碳/费用）；「—」的语义解释清楚（无数据不编造）；内部代号 S1/S2 不出现（转译为「联网的桩/未联网的桩」）；能量流真值需 CT 互感器的限定保留。
5. **通道与在线口径沿用权威定义**——「蓝牙近场/云端远程」固定词、「蓝牙能连上即在线」判定、「无法确认≠失败」措辞，与帮助中心 connectivity 章节逐句对齐。

## 四、本次写作中确认的口径取舍（后续维护须知）

- 预约模式命名：帮助中心用「单次/每周重复」、销售手册用「一次性/每周循环」、**App 实际 UI 用「一次性/循环」**——说明书从 App。若后续统一，建议反向修帮助中心。
- Wi-Fi 频段：帮助中心留 TODO 未断言，但 **App 配网页明示「设备仅支持 2.4 GHz WiFi 网络。」**——说明书如实引用 App 文案（描述 App 行为，不替产品拍板协议口径）。
- 支持邮箱：帮助中心 TODO 未定——说明书不写邮箱，只指向 App 内反馈入口。
- 推送通知：当前版本点按推送**无跳转落点**（未实现 tap-through）——说明书未写「点通知进入消息中心」。
- 「关于此充电桩」与「关于（App）」是两个不同页面，说明书分别放在 §11.2 与 §12.4，不可合并。
- 昵称/头像为纯本机存储（页面底部「展示给共享成员」文案与实现不符）——说明书按用户可感知行为写「仅保存在本机」。

## 五、维护流程

- 功能变更时先改本目录权威源，再同步派生出口；
- 界面文案变更以 `Localizable.strings` diff 为触发信号，逐「」核对；
- 若落地 App 内嵌出口，权威源迁移规则与帮助中心相同（server 仓为权威、本目录降级镜像），迁移前以本目录为准。
